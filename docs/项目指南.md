# AgentSwarm：企业级多智能体操作系统全景详解

## 一、 项目背景与终极愿景

在当前的大模型应用领域，多智能体（Multi-Agent）框架普遍面临两大痛点：一是**缺乏底层物理隔离**，多个 Agent 往往运行在同一个进程中，极易产生状态污染；二是**业务逻辑编排薄弱**，难以处理长周期、多分支的复杂协作。

**AgentSwarm** 的诞生正是为了解决这一核心矛盾。本项目创造性地融合了三大前沿理念，旨在构建一个“从底层基础设施到上层业务逻辑全覆盖”的下一代 AI 操作系统：
1.  **Paperclip 的灵魂**：引入组织级编排，将宏观业务目标（Goal）动态拆解为具体任务（Issue），并实施严格的预算与状态对齐。
2.  **Claworc 的骨架**：依托底层物理基础设施管控能力，为每一个 Agent 提供独立的容器隔离与面板级操作体验。
3.  **Hermes 的大脑**：以在工具调用（Tool Calling）和结构化输出上表现优异的 Hermes 大模型为核心执行引擎，原生兼容 MCP（Model Context Protocol）协议。

## 二、 核心架构蓝图 (Top-Level Architecture)

AgentSwarm 采用“控制面-数据面”相分离的经典分布式架构，自上而下划分为三个逻辑平面：

### 1. 控制平面 (Control Plane) —— 系统的中枢神经
基于 **FastAPI + MongoDB** 构建，负责全局调度、状态流转与资源分配。
* **安全 API 网关 (Gateway)**：统一接管所有 Worker 发往大模型的请求。实现精准的 Token 计费打点，确保 API Key 零泄露。
* **状态机与智能合约**：维护有向无环图（DAG）形式的工作流状态，记录每个 Issue 的生命周期与重试次数。
* **Swarm 调度引擎**：直接与底层的 Docker Swarm API 通信，负责容器的动态拉起、暂停与销毁。

### 2. 执行与基础设施平面 (Execution Plane) —— 物理级的隔离前线
基于 **Docker Swarm** 构建，是 Agent 真正执行任务的沙箱网络。
* **Worker 节点群**：封装了标准化 Hermes 运行环境的独立 Docker 容器。
* **项目工作空间 (Project Workspace)**：基于 Swarm Volume 实现的跨节点共享数据卷，作为 Agent 协作的“物理共享白板”。

### 3. 交互与观测平面 (Observability Plane) —— “三位一体”监控台
为人类管理员提供强大的开发者体验（DX）与全局掌控力。
* **动态拓扑网 (React Flow)**：实时渲染系统内的 Agent 节点连线与状态呼吸灯。
* **沙箱编辑器 (Monaco Editor)**：类似 VS Code 的网页视图，提供实时文件锁显示与代码/文本 Diff 比对。
* **全链路监控器**：透视底层的 Token 消耗与大模型结构化 JSON 输出日志。

## 三、 四大关键流转机制

AgentSwarm 放弃了脆弱的单体并发，采用了一套极其严密且支持纠错的工程化流转机制：

### 1. 纯 AI 动态编排 (Auto-Orchestration)
摒弃静态 YAML 连线。系统存在一个具备“上帝视角”的 **全局 Coordinator Agent**。它能根据用户的宏大愿景，动态调用 `create_issue` 和 `spawn_worker_agent` 工具，实时向 Swarm 集群申请计算资源并赋予新建容器特定的人格与技能。

### 2. 严格流水线与独占式沙箱 (Strict Pipeline)
针对如“故事工厂”这类复杂内容生成场景，采用串行工作流。同一时刻，FastAPI 网关只允许唯一一个拥有执行权的 Agent 对项目白板发起读写。流转交接时，底层自动对 Volume 目录执行增量快照备份（Snapshot）。

### 3. 逆流重定向与强力纠错 (Rollback & Context Injection)
当后续节点（如审查 Agent）发现前置节点（如生成 Agent）的逻辑漏洞时：
* 触发 `reject_to_previous_node` 结构化驳回指令。
* 网关强制剥夺当前控制权，将物理沙箱**精准回滚**至出错前的一刻。
* 再次拉起生成 Agent 时，网关会将驳回原因作为 `[CRITICAL SYSTEM FEEDBACK]` 强行注入其 Prompt，打破大模型固执己见的幻觉死循环。

### 4. 时停审批流 (Human-in-the-Loop)
在修改核心设定、大规模拉起资源等关键事务上，系统将触发挂起（Suspend）机制。Agent 容器进入休眠，Web 拓扑图亮起警报。人类管理员审阅 Diff 后，可一键放行（Approve）、带意见驳回（Reject），甚至通过 Web 终端直接接管沙箱手动修改。

## 四、 扩展生态：内置技能注册中心 (Skill Registry)

为了让系统具备无限的横向扩展能力，AgentSwarm 内置了强类型约束的技能兵工厂。

* **声明式开发**：通过原生 Python 函数、Pydantic 类型注解以及 `@AgentSkill` 装饰器，开发者只需关注业务逻辑。系统会自动利用 AST 将其反向解析为 Hermes 能理解的 MCP 工具 Schema。
* **语义化检索**：全局 Agent 遭遇未知难题时，可根据意图动态搜索注册中心，实现工具的“即想即配”。
* **热挂载分发**：支持将含有 `requirements.txt` 和 `skillpack.yaml` 的技能包目录，在拉起 Worker 容器瞬间以只读 Volume 形式动态挂载，实现毫秒级技能装备。

## 五、 测试策略与质量保证

针对多智能体系统的非确定性，AgentSwarm 采用四层分离测试体系：
1.  **Mock 单元测试**：剥离大模型，向网关注入极端报错 JSON，校验框架底层状态机与计费逻辑的鲁棒性。
2.  **基础设施集成测试**：在真实 CI 环境中运行轻量脚本，验证 Docker Swarm 卷挂载与并发锁的隔离性。
3.  **属性校验 (LLM-as-a-Judge)**：不校验文本绝对值，由廉价大模型作为裁判，断言 Hermes 结构化输出是否符合 Schema 约束。
4.  **幽灵玩家 E2E 模拟**：通过自动化脚本监听 WebSocket，模拟人类触发并处理 `PENDING_APPROVAL` 挂起事件，验证全链路流转。

## 六、 实施路径规划 (Milestones)

系统以打造可落地的“故事工厂”（如《星际农业大亨》复杂设定生成验证）为核心驱动标杆：

* **Phase 1: 基础设施筑基** —— 跑通 FastAPI 网关、单体 Docker Swarm 容器拉起以及基础沙箱文件读写（MCP 工具挂载）。
* **Phase 2: 流水线与扩展中心** —— 实现状态机闭环、快照回滚纠错机制，上线基于 Pydantic 的 `@AgentSkill` 注册中心。
* **Phase 3: 动态编排与交付** —— 赋予全局 Agent 拉起新 Worker 的特权，跑通完整的时停人工审批流，交付 Pre-Alpha 演示原型。
* **Phase 4: 开源社区建设** —— 发布 Web 监控控制台，完善 `docker-compose` 本地一键部署方案，建立公共 Skill Hub。
