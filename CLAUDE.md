# 基本原则

## 1. 编码前先思考

**不要主观臆断。不要掩饰困惑。公开权衡取舍。**

在开始实现之前：
- 明确陈述你的假设。如果不确定，请提问。
- 如果存在多种理解方式，请把它们都列出来——不要默默地只选其中一种。
- 如果有更简单的实现方法，请提出来。在有正当理由时，要敢于提出反对意见。
- 如果有任何不清晰的地方，请停下来。指明哪里令人困惑，然后提问。

## 2. 简单至上

**用最少的代码解决问题。拒绝任何推测性代码。**

- 不要添加任何要求之外的功能。
- 不要为一次性使用的代码创建抽象。
- 不要提供未被要求的“灵活性”或“可配置性”。
- 不要为不可能发生的场景编写错误处理。
- 如果你写了 200 行代码，但本来只需 50 行就能搞定，请重写它。
- 实现方案必须简洁，不得过度工程化

问问你自己：“高级工程师会认为这太复杂了吗？”如果是，请简化。

## 3. 外科手术式的精准修改

**只触碰必须修改的地方。只清理你自己制造的烂摊子。**

修改现有代码时：
- 不要去“改进”相邻的代码、注释或格式。
- 不要重构没有出问题的东西。
- 匹配现有的代码风格，即使你个人倾向于不同的写法。
- 如果你注意到无关的死代码，提出来就好——不要删除它。

当你的修改产生了孤立（未使用）的代码时：
- 移除因**你**的修改而变得不再使用的导入（imports）、变量或函数。
- 除非被要求，否则不要移除原本就已经存在的死代码。

检验标准：每一行被修改的代码都应该能直接追溯到用户的请求。

## 4. 目标驱动的执行

**定义成功标准。循环执行直至验证通过。**

将任务转化为可验证的目标：
- “添加验证” → “为无效输入编写测试，然后让测试通过”
- “修复 Bug” → “编写一个能复现该 Bug 的测试，然后让测试通过”
- “重构 X” → “确保在重构前后测试都能通过”

对于多步骤的任务，列出一个简要的计划：
```text
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

强有力的成功标准能让你独立完成循环验证。薄弱的标准（如“让它能跑起来”）则需要不断地澄清。
